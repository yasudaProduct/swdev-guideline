# プロジェクトガイドライン整備 全体方針

最終更新: 2026-04-28（第1弾の観点リスト全てが完成）
ステータス: 第1弾 観点リスト完成（基本設計・詳細設計、全特性）

## 1. 目的

### 背景

本部レベルでプロジェクトガイドラインは存在し、提案から完了報告までの大枠（レビュー体系、必要書類など）は定められている。しかし、本部ガイドラインはあらゆるプロジェクトに適用できる汎用的な内容に留まっており、実際のプロジェクト遂行で必要となる具体的な判断基準やテンプレートは各プロジェクトの裁量に委ねられている。

その結果、以下の問題が発生している。

- プロジェクト間で品質にばらつきが生じている
- レビュー観点が各プロジェクトで個別に作成されており、レビュアーの負担が大きい
- スキルの浅いメンバーにとって、何を満たせば良いのかが事前にわかりにくい

### 目的

本部ガイドラインの下位に位置づく **具体ガイドライン群** を整備することで、プロジェクト遂行の品質を底上げし、個人スキル依存を減らし、レビューや設計の作業負担を軽減する。

## 2. 扱う範囲

### 今回の整備対象（メイン）

第1層：プロジェクト遂行の品質

第1層の中で、以下を扱う対象とする。

- レビュー観点テンプレート ← **第1弾として着手**
- プロジェクト特性別の進め方（改修・リプレイス・新規）
- 設計者・PG等のスキル差を埋める仕組み
- 設計書の品質基準

### 将来扱う可能性がある領域

第2層：成果物・ナレッジの資産化

- 設計書とソースの乖離問題
- プロジェクト間でのナレッジ横展開の仕組み
- 既存システム資料の維持・更新

### 当面の対象外

第3層：マネジメント可視化

- 報告フォーマットの統一、報告作業の効率化
- 理由：チーム移動による前提変更の可能性があるため、状況が落ち着くまで対象外とする

## 3. ガイドラインの位置づけ

### 本部ガイドラインとの関係

- 本部ガイドライン：上位ルール（大枠、必須）
- 本取り組みのガイドライン群：本部ガイドラインの **下位の具体ガイドライン**（本部ガイドラインを補完する位置づけ）

### 強制力

- すべての成果物について **「推奨」** とする
- プロジェクトの実情（コスト、難易度、特性など）に応じて、取捨選択や調整が可能
- 各プロジェクトのPMまたはレビュアーの判断で適用範囲を決める

### 適用範囲

- 当面、整備主導者の所属チーム内のプロジェクトを対象とする
- 運用と成果を踏まえて、将来的な適用範囲拡大は上長判断とする

## 4. 全体ロードマップ

### 第1弾（確定）：レビュー観点テンプレート

- 対象工程：基本設計、詳細設計
- 第1弾完了後に、他工程（要件定義、実装、各種テスト等）への展開を検討

### 第2弾以降の候補（順不同）

| 候補 | 内容 |
|------|------|
| 候補A | プロジェクト特性別の進め方ガイド（改修・リプレイス・新規） |
| 候補B | 設計書の品質基準 |
| 候補C | 設計者・PG等のスキル差を埋める仕組み |
| 候補D | レビュー観点テンプレートの他工程への展開 |
| 候補E | レビュー観点テンプレートのソフトウェア種別固有部分（WEB・バッチ・API等） |

### 順序の考え方

- 第1弾の進行と運用結果を踏まえて、第2弾以降の優先順位を決める
- 候補B（設計書品質基準）は、第1弾のレビュー観点テンプレートと内容的に重なる部分が多いため、第1弾と並行または直後の着手が自然な可能性がある

## 5. 体制と運用

### 整備の体制

| 役割 | 担当 |
|------|------|
| 整備の主導（叩き台作成） | 私 |
| レビュー | 上長および他メンバー |
| 展開 | 上長 |

スケジュールおよびレビュー段取りは、整備主導者の裁量で実施する。

### 運用上の役割

ガイドライン群が整備された後、運用に必要となる役割を以下に定義する。
具体的な担当者の割り当ては保留。実態として複数役割を兼任する可能性がある。

| 役割 | 内容 |
|------|------|
| オーナー | ガイドライン全体の最終責任者。重要な変更の最終承認者。 |
| メンテナー | 日常的な更新作業を行う担当者。誤字修正、軽微な追記など。 |
| レビュアー | 改訂内容をレビューする立場の人。複数人想定。 |
| 問い合わせ窓口 | 利用者からの解釈質問・改善要望の受付役。 |
| 利用者 | ガイドラインを実際に使うレビュアー・設計者。 |

### 成果物の保管・管理

- **保管場所**：ファイルサーバ（具体的なパスは保留、当面は整備主導者のローカルフォルダで作業）
- **バージョン管理**：成果物内に改訂履歴を設ける
- **アクセス権**：誰でも閲覧・編集可能

### 運用上の未決事項

以下は第1弾の成果物が具体化した後に決定する。

- 更新の責任者と更新タイミング
- 改善のサイクル（フィードバックの収集と反映方法）
- ガイドラインを使う側の運用ルール

## 6. 成功の定義（活動全体として）

ガイドライン整備という活動全体としての成功基準を以下に定める。
判定は実感ベースで行い、定量的な目標は設定しない。

### 短期（第1弾完了時点）

- 基本設計・詳細設計のレビュー観点テンプレートが整備され、チーム内のプロジェクトで実際に活用されている
- レビュアーから「観点作成の負担が減った」「指摘の精度が上がった」という実感がある
- 設計者から「事前に何を見られるかわかって作業しやすい」という反応がある

### 中期（第2弾以降を経て）

- 第1層（プロジェクト遂行品質）の主要領域でガイドライン群が整備され、チーム内のスタンダードとして定着している
- スキルの浅いメンバーが入っても、ガイドラインを参照することで一定の品質を担保できる
- プロジェクト間で品質のばらつきが目に見えて減っている

### 長期（活動全体として）

- ガイドラインが「生きた文書」として更新され続けている（作って終わりになっていない）

## 7. 開発スタイルと前提

- 開発スタイル：**ウォーターフォールに限定**
- プロジェクト規模：平均2〜5人、製造フェーズで5〜7人程度
- プロジェクト特性：改修案件、リプレイス案件、新規案件が混在

## 8. 第1弾のゴール（レビュー観点テンプレート）

### 主たるゴール

レビュー観点テンプレートを整備することで、レビューの質を底上げし、レビュアーの負担を減らす。

### 具体的に目指す状態

- レビュアーが観点作成にかける時間が減る
- 共通観点については、プロジェクト間で指摘内容のばらつきが減る
- スキルの浅い設計者でも、何を満たせばOKなのかが事前にわかる
- レビューで見落とされる典型的な問題が減る
- 後工程やレビュアーから設計者へのフィードバックが具体的になる

### 設計上の重要な前提

- **「共通観点」と「プロジェクト固有観点」を分離した構造** にする
  - テンプレートは共通観点を提供し、プロジェクト固有観点はレビュアーが追加する
- **テンプレートの利用者はレビュアーだが、設計者にも事前共有する**
  - 設計者がセルフチェックに使える
  - 設計者が「何を見られるか」を意識して作業できる（手戻り削減）
  - 暗黙知に頼らず、観点の明文化が促される

### 第1弾の構造（決定済み部分）

**観点の階層構造**

| 階層 | 内容 | 第1弾の対象 |
|------|------|-------------|
| 全プロジェクト共通 | 特性によらず、すべてのプロジェクトに適用される観点 | ○ |
| プロジェクト特性別 | 改修・リプレイス・新規の各特性に固有の観点 | ○ |
| ソフトウェア種別固有 | WEB・バッチ・API等、ソフトウェアの種類に固有の観点 | × （候補Eへ） |
| プロジェクト固有 | 各プロジェクトでレビュアーが追加する個別観点 | × （テンプレート対象外） |

**観点の分類軸**

- メイン軸：設計書の章立てに沿った分類
- サブ軸：観点カテゴリ（機能性・性能・保守性・セキュリティ・整合性・形式/体裁等）

**ソフトウェア種別固有部分の扱い**

- 第1弾では「全プロジェクト共通」の本体テンプレートを作成する
- ソフトウェア種別固有（WEB/バッチ/API等）の観点は、本体とは別建ての追加ドキュメントとして、第2弾以降（候補E）で段階的に整備する
- 理由：実際にそのソフトウェア種別を担当している人が知見を持っているため、後から知見保有者が追加する形が現実的

**基本設計と詳細設計の関係**

- 基本設計用と詳細設計用で **独立した2つのテンプレート** として作成する
- 両工程で観点の粒度・視点が異なる前提（基本設計は要件・業務との整合中心、詳細設計は実装可能性・保守性中心など）
- 共通部分（誤字脱字、用語統一など形式系の観点）は、両テンプレート作成後にボトムアップで抽出し、別ファイルに切り出す

**テンプレートの物理形式**

- 最終形式：**Excel（1ファイルで完結する形）**
- 列構成：観点ID、設計書の章、観点カテゴリ、観点名、観点の解説、判断基準、チェック欄、コメント欄
- ファイル単位：基本設計用、詳細設計用で別ファイル
- **作成段階では Markdown形式で作業し、内容確定後にExcelへ変換する**

## 9. 進め方の骨格

1. ゴールと成功の定義 ← **完了**
2. スコープと優先順位の決定 ← **完了**
3. 成果物の構造設計 ← **完了**
4. 各工程のレビュー観点を実際に作る ← **完了（Markdown版）**
5. 運用・改善の仕組み

次の作業：Excel化、上長・メンバーへのレビュー
