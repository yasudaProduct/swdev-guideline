# 1章：概要・目的

## 観点ID: BS-RPL-001
- **設計書の章**: 概要・目的
- **観点カテゴリ**: 機能性
- **観点名**: リプレイスの背景・目的が明確に記述されているか
- **観点の解説**:
  なぜリプレイスを行うのか（老朽化、サポート切れ、技術的負債、性能限界、コスト等）が明確に記述されているかを確認する。
  背景が不明確だと、リプレイス範囲や採用技術の妥当性判断ができない。
- **判断基準**:
  - リプレイスのきっかけ（既存システムの問題、外部要因等）が記述されている
  - リプレイスの目的（何を達成したいか）が記述されている
  - リプレイスによって解決される課題が示されている

## 観点ID: BS-RPL-002
- **設計書の章**: 概要・目的
- **観点カテゴリ**: 機能性
- **観点名**: リプレイスの方針（範囲・粒度）が明確に示されているか
- **観点の解説**:
  リプレイスの範囲（システム全体か一部か）と、機能・業務をどこまで踏襲しどこから刷新するかの方針が示されているかを確認する。
  方針が曖昧だと、後工程で「これは踏襲か、刷新か」の判断が都度発生し、設計が揺れる。
- **判断基準**:
  - リプレイス対象システム・サブシステムの範囲が示されている
  - 機能・業務をそのまま踏襲する範囲が示されている
  - 刷新（業務変更、機能改善等）を含む範囲が示されている
  - 既存システムを残す部分（連携先として継続利用する等）が示されている

## 観点ID: BS-RPL-003
- **設計書の章**: 概要・目的
- **観点カテゴリ**: 機能性
- **観点名**: 既存システムの調査範囲と方針が示されているか
- **観点の解説**:
  リプレイス元の既存システムについて、何を調査するか（機能、業務、データ、運用、連携等）の方針が記述されているかを確認する。
  既存調査の方針がないと、調査漏れによってリプレイス後に既存機能の欠落が発覚する。
- **判断基準**:
  - 調査対象（機能、画面、帳票、I/F、バッチ、データ、運用手順等）の範囲が示されている
  - 既存ドキュメントとソースのどちらを正として調査するかの方針が示されている
  - 既存システム関係者へのヒアリング計画が示されている、または参照先が示されている

## 観点ID: BS-RPL-004
- **設計書の章**: 概要・目的
- **観点カテゴリ**: 整合性
- **観点名**: 旧システムと新システムの並行稼働・切り替え方針が示されているか
- **観点の解説**:
  旧システムから新システムへの切り替え方法（一括切替、段階的切替、並行稼働等）の基本方針が、概要レベルで示されているかを確認する。
  切り替え方針は他の章にも影響する重要な前提なので、概要・目的の章で位置づけを明確にする必要がある。
- **判断基準**:
  - 切り替え方式（一括切替、段階切替、並行稼働等）の方針が示されている
  - 並行稼働がある場合、その期間と運用方針が記述されている
  - 詳細は別章で扱う場合、その参照先が示されている

# 2章：システム構成

## 観点ID: BS-RPL-005
- **設計書の章**: システム構成
- **観点カテゴリ**: 整合性
- **観点名**: 旧システムと新システムの構成の対応関係が示されているか
- **観点の解説**:
  旧システムの構成要素（サーバ、ミドルウェア、機能ブロック等）が、新システムでどの構成要素に対応するかが整理されているかを確認する。
  対応関係が不明確だと、リプレイス後に「旧システムにあった機能が新システムのどこに行ったかわからない」という事態が生じる。
- **判断基準**:
  - 旧システムと新システムの構成要素の対応表が示されている
  - 対応関係が変わる箇所（統合、分離、廃止、新設）が明示されている
  - 廃止される旧構成要素について、その機能の代替先が示されている

## 観点ID: BS-RPL-006
- **設計書の章**: システム構成
- **観点カテゴリ**: 信頼性
- **観点名**: 並行稼働期間中のシステム構成が定義されているか
- **観点の解説**:
  旧システムと新システムが並行稼働する期間において、両システムの構成・連携・運用がどうなるかが定義されているかを確認する。
  並行稼働期間の構成検討が不足すると、二重運用の負荷や、データ不整合のリスクが生じる。
- **判断基準**:
  - 並行稼働期間中のシステム構成図が示されている
  - 旧システムと新システムの間のデータ同期・連携方式が定義されている
  - 利用者がどちらのシステムを使うかのルールが定義されている
  - 並行稼働期間の運用負荷の見積りが記述されている

## 観点ID: BS-RPL-007
- **設計書の章**: システム構成
- **観点カテゴリ**: 機能性
- **観点名**: 旧システムの廃止・撤去計画が示されているか
- **観点の解説**:
  リプレイス完了後の旧システムの扱い（廃止、停止、保管等）と、その時期・手順が示されているかを確認する。
  撤去計画がないと、不要な旧システムが残り続け、コスト増・運用混乱・セキュリティリスクの原因になる。
- **判断基準**:
  - 旧システム廃止のタイミングと手順が示されている
  - 旧データの保管・廃棄方針が定義されている
  - 旧システムのライセンス・契約の整理が計画されている
  - 撤去後の影響（連携先への通知、運用手順の更新等）が考慮されている

# 3章：データ設計

## 観点ID: BS-RPL-008
- **設計書の章**: データ設計
- **観点カテゴリ**: 整合性
- **観点名**: 旧データと新データのマッピングが示されているか
- **観点の解説**:
  旧システムのデータ構造と新システムのデータ構造の対応関係（マッピング）が示されているかを確認する。
  マッピングが不十分だと、データ移行時にデータの欠落・重複・誤変換が発生する。
- **判断基準**:
  - 旧テーブル・旧項目と新テーブル・新項目の対応表が示されている
  - 1対1で対応しない箇所（統合、分割、変換、廃止）が明示されている
  - 旧データのコード値と新データのコード値の対応が示されている
  - マッピングできない（移行不要・移行不能）データが明示されている

## 観点ID: BS-RPL-009
- **設計書の章**: データ設計
- **観点カテゴリ**: 機能性
- **観点名**: データ構造の刷新方針が明確になっているか
- **観点の解説**:
  リプレイスにあたって、データ構造を旧システムからどの程度刷新するか（そのまま踏襲、正規化、論理モデル変更等）の方針が記述されているかを確認する。
  刷新方針が曖昧だと、設計が中途半端になり、リプレイスの利点を活かせない。
- **判断基準**:
  - データ構造の踏襲・刷新の方針が記述されている
  - 刷新する場合、刷新の根拠（業務改善、性能改善、保守性向上等）が示されている
  - 旧構造を踏襲する箇所と、その理由が示されている

## 観点ID: BS-RPL-010
- **設計書の章**: データ設計
- **観点カテゴリ**: 信頼性
- **観点名**: 移行データの品質確保方針が定義されているか
- **観点の解説**:
  旧システムから新システムへ移行するデータについて、品質（整合性、欠損、不正値等）を確保する方針が定義されているかを確認する。
  移行元データに品質問題があるまま移行すると、新システムに問題が引き継がれる。
- **判断基準**:
  - 移行元データの品質チェック観点が定義されている
  - 不正データ・欠損データの扱い（補正、除外、エラー扱い等）が記述されている
  - データクレンジングの方針と実施タイミングが示されている
  - 移行後のデータ検証方針が示されている

# 4章：機能設計

## 観点ID: BS-RPL-011
- **設計書の章**: 機能設計
- **観点カテゴリ**: 整合性
- **観点名**: 旧機能と新機能の対応関係が示されているか
- **観点の解説**:
  旧システムの機能一覧と新システムの機能一覧の対応関係が示されているかを確認する。
  対応関係が不明確だと、リプレイス後に「旧システムにあった機能が新システムにない」という事態が発生する。
- **判断基準**:
  - 旧機能と新機能の対応表が示されている
  - 1対1で対応しない箇所（統合、分割、廃止、新設）が明示されている
  - 廃止する旧機能について、廃止理由と代替手段が記述されている
  - 新規追加する機能と、その追加根拠が示されている

## 観点ID: BS-RPL-012
- **設計書の章**: 機能設計
- **観点カテゴリ**: 機能性
- **観点名**: 旧機能の隠れた仕様（暗黙仕様）の調査が行われているか
- **観点の解説**:
  旧システムには設計書に記載されていない暗黙の仕様（業務上の運用ルール、過去の不具合対応の名残、特殊ケースの処理等）が存在することが多い。これらが調査され、新システムへの反映方針が決まっているかを確認する。
  暗黙仕様の調査漏れは、リプレイス後の業務影響として現れやすい。
- **判断基準**:
  - 暗黙仕様の調査方法（ソースレビュー、業務ヒアリング、データ調査等）が記述されている
  - 発見された暗黙仕様の一覧が記録されている
  - 各暗黙仕様について、新システムへの反映方針（踏襲、廃止、変更）が定義されている

# 5章：画面・帳票・インターフェース設計

## 観点ID: BS-RPL-013
- **設計書の章**: 画面・帳票・インターフェース設計
- **観点カテゴリ**: 整合性
- **観点名**: 旧画面・帳票・I/Fと新画面・帳票・I/Fの対応関係が示されているか
- **観点の解説**:
  旧システムの画面・帳票・I/Fと、新システムのそれらの対応関係が示されているかを確認する。
  対応関係が不明確だと、利用者が新システムでどの画面・帳票を使えばよいかわからず、業務が滞る。
- **判断基準**:
  - 旧画面・旧帳票・旧I/Fと新側の対応表が示されている
  - 1対1で対応しない箇所（統合、分割、廃止、新設）が明示されている
  - 廃止する旧画面・旧帳票について、業務上の代替手段が記述されている

## 観点ID: BS-RPL-014
- **設計書の章**: 画面・帳票・インターフェース設計
- **観点カテゴリ**: 機能性
- **観点名**: 利用者の操作感・帳票フォーマットの踏襲方針が定義されているか
- **観点の解説**:
  リプレイスにあたって、利用者の操作感（画面レイアウト、操作手順）や帳票フォーマットを旧システムから踏襲するか刷新するかの方針が記述されているかを確認する。
  方針が曖昧だと、利用者の混乱や、業務マニュアルの大幅改訂が必要になる。
- **判断基準**:
  - 画面・帳票の踏襲・刷新の方針が示されている
  - 刷新する場合、刷新の根拠(UX改善、技術的制約等)が記述されている
  - 利用者教育・マニュアル更新の必要性と方針が記述されている

## 観点ID: BS-RPL-015
- **設計書の章**: 画面・帳票・インターフェース設計
- **観点カテゴリ**: 整合性
- **観点名**: 既存I/Fの連携先への影響と移行方針が定義されているか
- **観点の解説**:
  リプレイスに伴い、既存の外部I/Fを刷新するか、現行仕様を維持するかの方針が、連携先との合意を含めて記述されているかを確認する。
  リプレイスでは外部I/F全体の刷新も検討されるが、連携先側の都合で旧仕様維持が必要な場合もあり、調整が論点となる。
- **判断基準**:
  - 既存I/Fごとに刷新・維持の方針が示されている
  - 連携先との合意状況が記述されている
  - 並行稼働期間中のI/F運用方針が定義されている
  - 連携先の対応スケジュール調整状況が記述されている

# 6章：外部システム連携

## 観点ID: BS-RPL-016
- **設計書の章**: 外部システム連携
- **観点カテゴリ**: 整合性
- **観点名**: 旧システムが行っていた連携の網羅的な把握が行われているか
- **観点の解説**:
  旧システムが行っているすべての外部システム連携（定常連携・突発連携・例外時連携を含む）が網羅的に把握されているかを確認する。
  連携の把握漏れは、リプレイス後に「予期せぬタイミングで使われていた連携」が動かない事態を招く。
- **判断基準**:
  - 旧システムの全連携が一覧化されている
  - 連携の発生頻度（毎日・毎月・年次・突発等）が把握されている
  - 例外時にしか動かない連携も識別されている
  - 文書化されていない連携（手作業を含む）の調査状況が記述されている

## 観点ID: BS-RPL-017
- **設計書の章**: 外部システム連携
- **観点カテゴリ**: 信頼性
- **観点名**: 並行稼働期間中の連携データの一貫性確保方針が定義されているか
- **観点の解説**:
  旧システムと新システムが並行稼働する期間中、外部システムとの連携データが両システム間で一貫性を保てるかが定義されているかを確認する。
  並行稼働期間中の連携不整合は、外部システム側のデータ汚染を引き起こす。
- **判断基準**:
  - 並行稼働期間中、どちらのシステムが連携を行うかのルールが定義されている
  - 両システムから同一連携先へ連携する場合の重複防止策が記述されている
  - 連携先システムへ並行稼働の事実と影響を通知する計画が示されている
  - 並行稼働終了後の連携切り替え手順が示されている

# 7章：性能・非機能要件の設計

## 観点ID: BS-RPL-018
- **設計書の章**: 性能・非機能要件の設計
- **観点カテゴリ**: 性能・効率性
- **観点名**: 旧システムの性能実績が新システムの性能要件の根拠となっているか
- **観点の解説**:
  リプレイスでは旧システムの性能実績（処理時間、応答時間、リソース使用量等）が把握されており、それが新システムの性能要件設定の根拠として活用されているかを確認する。
  旧実績を踏まえないと、新システムの性能要件が現実的でなかったり、過剰・過小になったりする。
- **判断基準**:
  - 旧システムの性能実績が把握・記録されている
  - 旧実績と新要件の対応関係が示されている
  - 性能を改善する箇所と、その目標値・根拠が記述されている
  - 性能を踏襲する箇所と、その理由が示されている

## 観点ID: BS-RPL-019
- **設計書の章**: 性能・非機能要件の設計
- **観点カテゴリ**: 信頼性
- **観点名**: 旧システムの稼働実績（可用性・障害履歴）が新システムの設計に活かされているか
- **観点の解説**:
  旧システムの稼働実績（過去の障害、停止頻度、障害原因等）が把握され、新システムの信頼性設計に活かされているかを確認する。
  旧実績を踏まえないと、過去に発生していた問題が新システムでも繰り返される。
- **判断基準**:
  - 旧システムの過去障害・問題事象が整理されている
  - 各事象に対する新システムでの対策方針が示されている
  - 旧システムで継続的に発生していた運用課題への対応方針が示されている

# 8章：セキュリティ設計

## 観点ID: BS-RPL-020
- **設計書の章**: セキュリティ設計
- **観点カテゴリ**: セキュリティ
- **観点名**: 旧システムのセキュリティ仕様の見直しが行われているか

# 独自章1：データ移行
