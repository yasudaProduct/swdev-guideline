## このファイルの位置づけ

第1弾「レビュー観点テンプレート」のうち、**詳細設計の全プロジェクト共通観点** を整備したリスト。

最終的にはExcel形式に変換するが、内容が固まるまではMarkdownで作業する。

## 観点フォーマット

```markdown
### 観点ID: [ID]
- **設計書の章**: [章名]
- **観点カテゴリ**: [カテゴリ]
- **観点名**: [観点名（疑問文）]
- **観点の解説**:
  [なぜこの観点が必要か、何を見るのかの説明]
- **判断基準**:
  - [判断基準1]
  - [判断基準2]
```

### 観点IDの命名規則

- 形式：`DD-COM-{連番}`（詳細設計・全プロジェクト共通）
- 連番：3桁

### 観点カテゴリ（8種、暫定）

| カテゴリ | 内容 |
|---------|------|
| 機能性 | 要件を満たす機能が正しく設計されているか |
| 整合性 | 設計内の他箇所、上位設計、既存システムとの整合 |
| 性能・効率性 | 処理性能、応答性能、リソース効率 |
| 信頼性 | エラー処理、例外処理、リカバリ、ログ |
| セキュリティ | 認証、認可、データ保護、脆弱性対策 |
| 保守性 | 変更容易性、可読性、ソースとの整合のしやすさ |
| テスト容易性 | テスト可能な設計になっているか |
| 形式・体裁 | 用語、命名、誤字脱字、書式 |

### 設計書の章立て（基本設計をベースに詳細設計向けに調整）

1. 概要・目的
2. システム構成
3. データ設計（テーブル物理設計・SQLを含む）
4. 機能設計（クラス設計・モジュール構造を含む）
5. 画面・帳票・インターフェース設計
6. 外部システム連携
7. 性能・非機能要件の設計
8. セキュリティ設計
9. エラー処理・例外処理
10. 運用・保守
11. 処理フロー詳細（新設）
12. テスト設計（新設）
- 全般：特定の章に紐づかない、文書全体に対する観点

# 1章：概要・目的

## 観点ID: DD-COM-001
- **設計書の章**: 概要・目的
- **観点カテゴリ**: 整合性
- **観点名**: 詳細設計書の位置づけと対象範囲が明確か
- **観点の解説**:
  この詳細設計書がどの基本設計書を詳細化したものか、どの範囲（機能、画面、サブシステム等）を対象とするかが明確に記述されているかを確認する。
  位置づけが不明確だと、設計書間の関係が追跡できず、保守時の調査効率が低下する。
- **判断基準**:
  - 対応する基本設計書の参照（ファイル名、版数、該当章）が明示されている
  - 詳細設計書の対象範囲（機能、コンポーネント等）が明示されている
  - 同一プロジェクト内の他の詳細設計書との関係が示されている

## 観点ID: DD-COM-002
- **設計書の章**: 概要・目的
- **観点カテゴリ**: 整合性
- **観点名**: 基本設計からの変更点が明示されているか
- **観点の解説**:
  詳細設計を進める中で基本設計の内容を変更・補正した場合、その変更点が明示され、基本設計書側との整合が取れる仕組みになっているかを確認する。
  変更が記録されないと、基本設計書と詳細設計書の乖離が発生する。
- **判断基準**:
  - 基本設計から変更した箇所と理由が記録されている
  - 基本設計書の修正が必要な箇所が明示されている
  - 基本設計書を修正しない場合、その判断と影響が記述されている

# 2章：システム構成

## 観点ID: DD-COM-003
- **設計書の章**: システム構成
- **観点カテゴリ**: 機能性
- **観点名**: 詳細レベルでの構成要素・配置が定義されているか
- **観点の解説**:
  基本設計のシステム構成を踏まえ、詳細レベルで必要となる構成要素（プロセス、サービス、ライブラリ、設定ファイル等）とその配置が定義されているかを確認する。
  詳細レベルでの構成が不明確だと、実装時にデプロイ・配置の判断ができない。
- **判断基準**:
  - 詳細レベルの構成要素（実行モジュール、設定ファイル、リソース等）が一覧化されている
  - 各構成要素の配置場所（サーバ、ディレクトリ等）が定義されている
  - 構成要素の起動・停止順序が必要な箇所で詳細化されている

# 3章：データ設計（テーブル物理設計・SQL含む）

## 観点ID: DD-COM-004
- **設計書の章**: データ設計
- **観点カテゴリ**: 機能性
- **観点名**: テーブル物理設計が定義されているか
- **観点の解説**:
  基本設計の論理データ設計を踏まえ、物理テーブル（テーブル名、カラム名、型、制約、デフォルト値等）が定義されているかを確認する。
  物理設計が曖昧だと、実装時にDDLが書けない、または恣意的な物理化が行われる。
- **判断基準**:
  - テーブル名・カラム名の命名規則に従った物理名が定義されている
  - 各カラムの物理型・桁数・NULL許容・デフォルト値が定義されている
  - 主キー・外部キー・一意制約・チェック制約が物理レベルで定義されている
  - テーブル空間・ストレージ設定等の物理パラメータが必要に応じて定義されている

## 観点ID: DD-COM-005
- **設計書の章**: データ設計
- **観点カテゴリ**: 性能・効率性
- **観点名**: インデックス設計が適切に行われているか
- **観点の解説**:
  各テーブルのインデックスが、想定されるアクセスパターン（検索条件、結合条件、ソート等）に対して適切に設計されているかを確認する。
  インデックス設計が不適切だと、性能要件未達や、逆に更新性能の劣化が発生する。
- **判断基準**:
  - 主要な検索・結合・ソートのアクセスパターンが整理されている
  - 各アクセスパターンに対するインデックス設計の根拠が示されている
  - インデックスによる更新性能への影響が考慮されている
  - 不要なインデックス（過剰、重複）が排除されている

## 観点ID: DD-COM-006
- **設計書の章**: データ設計
- **観点カテゴリ**: 機能性
- **観点名**: SQL設計の方針が示されているか
- **観点の解説**:
  詳細設計レベルで、主要なSQL（複雑なクエリ、性能影響の大きいクエリ、共通的に利用されるクエリ等）の設計方針が示されているかを確認する。
  SQL方針がないと、実装者ごとに異なるSQLが書かれ、保守性・性能のばらつきが生じる。
- **判断基準**:
  - 性能影響の大きいSQLが識別されている
  - 共通的に利用されるSQLパターン（検索、集計、更新等）の方針が示されている
  - SQL記述ルール（命名、フォーマット、コメント等）が定義されている、または参照先が示されている
  - ストアドプロシージャ・トリガーの利用方針が定義されている

## 観点ID: DD-COM-007
- **設計書の章**: データ設計
- **観点カテゴリ**: 信頼性
- **観点名**: トランザクション境界が明確に定義されているか
- **観点の解説**:
  各処理におけるトランザクション境界（BEGIN/COMMIT/ROLLBACKの範囲）が明確に定義されているかを確認する。
  トランザクション境界が不明確だと、データ不整合・デッドロック・性能問題の原因になる。
- **判断基準**:
  - 各処理のトランザクション境界が定義されている
  - 複数のトランザクションをまたぐ処理について、その妥当性と整合性確保の手段が記述されている
  - 分離レベル（Isolation Level）の選択と根拠が示されている
  - トランザクションの長時間化を避ける設計が考慮されている

# 4章：機能設計（クラス設計・モジュール構造を含む）

## 観点ID: DD-COM-008
- **設計書の章**: 機能設計
- **観点カテゴリ**: 機能性
- **観点名**: クラス・モジュールの構造が定義されているか
- **観点の解説**:
  基本設計の機能を実装するためのクラス・モジュール・関数の構造（責務分担、配置、命名等）が定義されているかを確認する。
  構造が不明確だと、実装者ごとにバラバラな構造になり、保守性が低下する。
- **判断基準**:
  - クラス・モジュールの一覧と責務が定義されている
  - クラス間・モジュール間の関係（継承、依存、利用）が示されている
  - 命名規則・配置規則（パッケージ・名前空間）が定義されている、または参照先が示されている
  - 共通化すべき処理（ユーティリティ、ベースクラス等）が識別されている

## 観点ID: DD-COM-009
- **設計書の章**: 機能設計
- **観点カテゴリ**: 保守性
- **観点名**: クラス・モジュールの責務が単一になっているか
- **観点の解説**:
  各クラス・モジュールの責務が単一で明確であり、過度に多くの役割を持っていないかを確認する。
  責務が肥大化したクラスは、変更時の影響範囲が大きくなり、保守性が低下する。
- **判断基準**:
  - 各クラス・モジュールの責務が一文で説明できる粒度になっている
  - 責務が肥大化したクラスがある場合、分割の必要性が検討されている
  - 異なる責務（画面表示・業務ロジック・データアクセス等）が混在していない
  - 責務分担の根拠が記述されている

## 観点ID: DD-COM-010
- **設計書の章**: 機能設計
- **観点カテゴリ**: 機能性
- **観点名**: 各機能のメソッド・関数のインターフェースが定義されているか
- **観点の解説**:
  各機能を実現するメソッド・関数のインターフェース（名前、引数、戻り値、例外）が定義されているかを確認する。
  インターフェースが不明確だと、実装者ごとに異なる呼び出し方が生まれ、結合時に問題が発生する。
- **判断基準**:
  - 主要なメソッド・関数のシグネチャ（名前、引数、戻り値型）が定義されている
  - 引数・戻り値の意味と取りうる値の範囲が記述されている
  - スローされる例外の種類と発生条件が記述されている
  - メソッドの事前条件・事後条件が必要に応じて記述されている

## 観点ID: DD-COM-011
- **設計書の章**: 機能設計
- **観点カテゴリ**: 機能性
- **観点名**: 機能の処理ロジックが実装可能な粒度で記述されているか
- **観点の解説**:
  基本設計の機能仕様を踏まえ、詳細レベルの処理ロジック（分岐、ループ、計算式、業務ルール等）が実装可能な粒度で記述されているかを確認する。
  ロジックが粗いと、実装者の解釈で揺れが生じる。
- **判断基準**:
  - 主要な処理ステップがフローチャート、擬似コード、文章等で記述されている
  - 条件分岐の判定式と分岐後の処理が明確に記述されている
  - 計算式・業務ルールが具体的に記述されている
  - 例外的なケース・境界条件への対応が記述されている

## 観点ID: DD-COM-012
- **設計書の章**: 機能設計
- **観点カテゴリ**: 保守性
- **観点名**: 共通化・再利用の方針が示されているか
- **観点の解説**:
  複数の機能で共通的に必要となる処理（バリデーション、日付計算、文字列操作、ログ出力等）を共通化する方針が示されているかを確認する。
  共通化の方針がないと、同様のロジックが各所に散在し、保守性が低下する。
- **判断基準**:
  - 共通化対象の処理が識別されている
  - 共通処理の配置・呼び出し方が定義されている
  - 既存の共通処理（ライブラリ、フレームワーク機能等）の利用方針が示されている
  - 共通化の判断基準（再利用回数、影響範囲等）が記述されている

# 5章：画面・帳票・インターフェース設計

## 観点ID: DD-COM-013
- **設計書の章**: 画面・帳票・インターフェース設計
- **観点カテゴリ**: 機能性
- **観点名**: 各画面・帳票・I/Fの詳細項目仕様が定義されているか
- **観点の解説**:
  基本設計の項目定義を踏まえ、詳細レベル（画面項目の表示位置・サイズ、入力制御、帳票の罫線・フォント、I/Fの正確なデータレイアウト等）まで定義されているかを確認する。
  詳細項目仕様が曖昧だと、実装者ごとに異なる見た目・動作になる。
- **判断基準**:
  - 画面項目について、配置・サイズ・色・初期値等が定義されている
  - 入力制御（IME、最大文字数、入力可能文字種等）が定義されている
  - 帳票項目について、配置・書式（罫線、フォント、桁揃え等）が定義されている
  - I/F項目について、データレイアウト（オフセット、長さ、エンディアン等）が定義されている

## 観点ID: DD-COM-014
- **設計書の章**: 画面・帳票・インターフェース設計
- **観点カテゴリ**: 機能性
- **観点名**: 画面遷移・操作の詳細が定義されているか
- **観点の解説**:
  基本設計の画面遷移を踏まえ、詳細レベルの操作仕様（ボタン押下時の挙動、キー操作、ダイアログ表示、フォーカス遷移等）が定義されているかを確認する。
  操作詳細が不明確だと、実装後にユーザビリティの問題が発覚しやすい。
- **判断基準**:
  - 各操作（ボタン押下、リンククリック、キー入力等）に対する挙動が定義されている
  - 確認ダイアログ・エラーダイアログの表示条件と内容が定義されている
  - フォーカス遷移・タブ順が定義されている
  - 連続クリック・多重送信等の防止策が定義されている

# 6章：外部システム連携

## 観点ID: DD-COM-015
- **設計書の章**: 外部システム連携
- **観点カテゴリ**: 機能性
- **観点名**: 連携の詳細仕様（プロトコル・データレイアウト）が定義されているか
- **観点の解説**:
  基本設計で合意された連携を実装可能なレベルまで詳細化（具体的なプロトコル仕様、リクエスト・レスポンスの完全なデータレイアウト、認証情報の受け渡し等）されているかを確認する。

- **判断基準**:
  - 通信プロトコル（HTTP/HTTPS、SOAP、ファイル転送等）の詳細が定義されている
  - リクエスト・レスポンスの完全なデータレイアウトが定義されている
  - 認証情報・セッション情報の受け渡し方法が詳細に記述されている
  - 連携先システムの仕様書との対応が明示されている

- **観点カテゴリ**: 性能・効率性

- **観点の解説**:

  - リソース（コネクション、スレッド、メモリ等）の管理方針が示されている
  - 性能目標値と詳細設計の整合性が確認されている

# 8章：セキュリティ設計

## 観点ID: DD-COM-017
- **設計書の章**: セキュリティ設計
- **観点カテゴリ**: セキュリティ
- **観点名**: 実装レベルのセキュリティ対策が具体化されているか
- **観点の解説**:
  基本設計のセキュリティ方針を踏まえ、実装レベルの具体的な対策（SQLインジェクション対策の具体実装、XSS対策、認証トークンの取り扱い等）が記述されているかを確認する。
  実装レベルが曖昧だと、実装者ごとに対策のばらつきが生じ、脆弱性の原因となる。
- **判断基準**:
  - インジェクション対策の実装方針（プレースホルダ利用、エスケープ処理等）が示されている
  - 認証情報・セッショントークンの実装上の取り扱い方針が示されている
  - 入力値検証の実装方針が示されている
  - 暗号化処理の実装方針（使用するライブラリ、鍵管理等）が示されている

## 観点ID: DD-COM-018
- **設計書の章**: セキュリティ設計
- **観点カテゴリ**: セキュリティ
- **観点名**: ログ出力時の機密情報の取り扱いが詳細に定義されているか
- **観点の解説**:

  実装段階でログに機密情報が混入することは、本番運用後の情報漏洩の典型的な原因となる。

  - ログ出力でマスキングすべき項目が一覧化されている
  - マスキング処理の実装方針が示されている

  - ログレベルごとの情報粒度の方針が示されている

## 観点ID: DD-COM-025
- **設計書の章**: テスト設計
- **観点カテゴリ**: テスト容易性
- **観点名**: 単体テストの設計方針が記述されているか
- **観点の解説**:
  詳細設計したクラス・モジュール・関数に対する単体テストの設計方針（テスト範囲、テスト観点、カバレッジ目標等）が記述されているかを確認する。
  単体テスト方針がないと、実装者ごとに異なる粒度でテストが書かれ、品質保証が不安定になる。
- **判断基準**:
  - 単体テストの対象範囲（クラス、メソッド）が定義されている
  - テスト観点（正常系、異常系、境界値等）が示されている

  - テストフレームワーク・ツールの方針が示されている

- **観点名**: 結合テストの設計方針が記述されているか
- **観点の解説**:
  複数のクラス・モジュール・サブシステムを結合したテストの設計方針（結合の単位、テストシナリオ、データ準備等）が記述されているかを確認する。

# 全般（特定の章に紐づかない観点）

## 観点ID: DD-COM-029
- **設計書の章**: 全般
- **観点カテゴリ**: 保守性
- **観点名**: 詳細設計の判断根拠が記述されているか
- **観点の解説**:

  - 採用しなかった案がある場合、不採用理由が記述されている
