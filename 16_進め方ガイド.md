## このファイルの位置づけ

第3弾「プロジェクト特性別の進め方ガイド（候補A）」として、改修・リプレイス・新規の各特性に応じたプロジェクトの進め方の留意点を整備した文書。

第1弾「レビュー観点テンプレート」、第2弾「設計書品質基準」「種別固有観点」とあわせて、プロジェクトを進める際のガイドラインとして利用する。

# 1章：このガイドの位置づけ

## 1-1. このガイドの目的

このガイドは、プロジェクトの **特性に応じた進め方** をまとめた読み物です。

第1弾・第2弾が「設計書のレビュー観点」「設計書の品質基準」「ソフトウェア種別固有の観点」という **項目の集合** だったのに対し、本ガイドは **プロジェクト全体の進め方** に焦点を当てています。

具体的には、

- プロジェクト立ち上げ時に、特性に応じた進め方の見通しを持てるようにする
- スキルの浅いメンバーでも、特性別の留意点を事前に学べるようにする
- 特性固有の典型的な失敗を回避できるようにする

ことを目的としています。

## 1-2. このガイドの読み方

本ガイドは大きく4部構成です。

- **第1部 共通編**：全特性に共通する進め方の基本
- **第2部 改修案件編**：既存システムへの機能追加・変更・修正の進め方
- **第3部 リプレイス案件編**：既存システムを新しい技術スタックに置き換える進め方
- **第4部 新規案件編**：ゼロから新しいシステムを構築する進め方

プロジェクトの立ち上げ時には、まず第1部を読み、次に該当する特性の章を読んでください。
プロジェクト進行中も、節目ごとに該当章を読み返すことで、留意点の見落としを防げます。

## 1-3. 強制力

本ガイドは **「推奨」** とします。プロジェクトの実情（コスト、難易度、規模等）に応じて、取捨選択や調整が可能です。プロジェクトのPMまたはリーダーの判断で適用範囲を決めてください。

## 1-4. 第1弾・第2弾との関係

本ガイドは、以下の文書とあわせて使うことで効果を発揮します。

| 文書 | 用途 |
|------|------|
| 第1弾：レビュー観点テンプレート | 設計書レビュー時のチェック項目 |
| 第2弾：設計書品質基準 | 設計者が設計書を書くときに守る基準 |
| 第2弾：種別固有観点（WEB/バッチ/API） | ソフトウェア種別ごとの追加観点 |
| 本ガイド：プロジェクト特性別 進め方ガイド | プロジェクト全体の進め方の留意点 |

各特性の章には、関連する観点・基準のIDへの参照を記載しています。

# 2章：プロジェクト特性とは

## 2-1. 3つの特性

本ガイドでは、プロジェクトを以下の3つの特性に分類します。

| 特性 | 定義 |
|------|------|
| **改修案件** | 既存システムに対して機能追加・変更・修正を行う案件 |
| **リプレイス案件** | 既存システムを新しい技術スタック・基盤に置き換える案件 |
| **新規案件** | ゼロから新しいシステムを構築する案件 |

## 2-2. 特性の判定基準

実際のプロジェクトでは、これらの特性が混在することがあります。例えば、

- 既存システムをリプレイスしつつ、一部機能を新規追加する → リプレイス＋一部新規
- 改修プロジェクトの途中で、システム全体の刷新が決まる → 改修＋部分リプレイス

このような場合、**主たる特性** に応じて本ガイドを参照しつつ、副次的に該当する特性の章も参考にしてください。

判定の目安は以下です。

- **改修案件**：既存システムの構造・機能の50%以上を維持する
- **リプレイス案件**：既存システムを置き換えるが、業務要件・機能はおおむね踏襲する
- **新規案件**：参照すべき既存システムが存在しない、または存在しても全く別物として作る

## 2-3. 特性ごとの難易度・リスクの傾向

特性ごとに、典型的なリスク・難易度の傾向があります。

| 特性 | 難易度 | 主なリスク |
|------|--------|-----------|
| 改修案件 | 中 | 既存への影響、影響範囲の見落とし、リグレッション |
| リプレイス案件 | 高 | データ移行、機能等価性、切り替え、長期間の業務継続 |
| 新規案件 | 中〜高 | 要件の曖昧さ、業務プロセスの構築、初期データ整備 |

これらの傾向を踏まえて、各特性の章で詳細な留意点を解説します。

# 3章：全特性に共通する進め方の基本

## 3-1. プロジェクトの工程

ウォーターフォール開発を前提とした、典型的な工程は以下です。

```
要件定義 → 基本設計 → 詳細設計 → 実装 → 単体テスト → 結合テスト → システムテスト → 受入テスト → リリース → 運用・保守
```

特性ごとに、各工程で重視すべきことが異なります。詳細は各特性の章で解説します。

## 3-2. 工程ごとの基本姿勢

特性によらず、各工程で以下の基本姿勢を守ることが重要です。

### 要件定義
- 業務側との認識合わせを優先する
- 「何を作るか」を曖昧にしない
- スコープの境界を明確にする

### 基本設計・詳細設計
- 後工程の担当者が困らない粒度で書く
- レビュー観点・品質基準（第1弾・第2弾）を活用する
- 重要な判断には根拠を残す

### 実装・テスト
- 設計書からの逸脱を最小限にする
- 逸脱が必要な場合、設計書に反映する
- テストデータ・テスト環境は早めに準備する

### リリース・運用
- リリース前に運用関係者と十分に共有する
- 切り戻し（ロールバック）の手順を準備する
- リリース直後の監視体制を強化する

## 3-3. プロジェクト規模に応じた調整

本ガイドの内容は、チーム規模 **2〜7名** のプロジェクトを前提としています。それより小規模・大規模なプロジェクトでは、以下のような調整が必要です。

### 1〜2名の小規模プロジェクト
- ドキュメント作成の負荷が大きいため、必要最低限に絞る
- レビューは1名で行うことが多くなるため、セルフチェックの仕組みを工夫する
- 属人化しやすいので、最低限のドキュメントは必ず残す

### 8名以上の大規模プロジェクト
- メンバー間の認識合わせがより重要になる
- ドキュメントの粒度・体裁の統一を徹底する
- 役割分担を明確にする

## 3-4. リスク管理の基本

特性によらず、プロジェクトの早期段階でリスクを洗い出し、対策を計画することが重要です。

### リスク洗い出しのタイミング
- プロジェクト立ち上げ時（特に重要）
- 各工程の節目（フェーズゲート）
- 想定外の事象が発生した時

### 典型的なリスクの分類
- **スコープリスク**：要件の追加・変更
- **スケジュールリスク**：作業遅延
- **品質リスク**：成果物の品質低下
- **技術リスク**：技術的な実現困難
- **体制リスク**：要員確保、スキル不足
- **外部要因リスク**：外部システム・連携先・利害関係者起因

各特性で、特に注意すべきリスクの傾向を以降の章で解説します。

# 5章：改修案件とは

## 5-1. 改修案件の定義

改修案件は、既存システムに対して機能追加・変更・修正を行う案件です。具体的には以下のようなプロジェクトが該当します。

- 既存機能への新機能追加
- 既存機能の仕様変更
- 不具合修正・障害対応の恒久対策
- 法令対応・制度変更への対応
- 性能改善・保守性向上のための改修

## 5-2. 改修案件の特徴

改修案件は、以下の特徴を持ちます。

- **既存の制約がある**：既存システムの構造・データ・運用が継続して存在する
- **影響範囲が見えにくい**：改修箇所と関連する既存箇所への影響を正確に把握する必要がある
- **既存への影響を最小化することが重要**：意図しない既存機能の変更は重大な問題を招く
- **既存資料の質が重要**：既存の設計書・ソースの整合状況が改修の難度を左右する

## 5-3. 典型的な改修案件のパターン

| パターン | 特徴 | 難易度 |
|---------|------|--------|
| 単機能の追加 | 既存機能への影響が限定的 | 低 |
| 複数機能の変更 | 影響範囲の調査が重要 | 中 |
| 共通処理の改修 | 全機能への影響評価が必須 | 高 |
| データ構造の変更 | 既存データへの影響、移行作業が必要 | 高 |
| 性能改善 | 既存実装の理解と再設計が必要 | 中〜高 |

# 6章：改修案件の進め方の基本フロー

## 6-1. 工程ごとの進め方

### 要件定義フェーズ

改修要件の確定と、影響範囲の初期把握を行います。

- 改修要件の整理（業務要件、不具合対応要件、改善要件等）
- 改修対象の概要レベルでの特定
- 改修によって解決される課題の明確化
- 業務側との合意（変更点、影響、利用者教育の必要性等）

### 基本設計フェーズ

改修内容の詳細化と、既存システムへの影響範囲の精査を行います。

- 既存設計書・既存ソースの整合状況の確認
- 改修箇所の特定と影響範囲調査（→ 第1弾 BS-MOD-021〜023）
- 既存機能への副作用の評価
- 既存仕様の維持・変更の業務側合意（→ BS-MOD-028）
- リグレッション防止方針の策定（→ BS-MOD-027）

### 詳細設計フェーズ

実装可能なレベルまで設計を具体化します。

- 既存実装との整合確認（→ DD-MOD-003）
- 共通処理の改修影響評価（→ DD-MOD-005）
- リファクタリング範囲の明確化（→ DD-MOD-004）
- 既存テスト資産の活用方針（→ DD-MOD-006）

### 実装フェーズ

設計書に基づいて実装を行います。

- 既存コードベースの規約に準拠した実装
- 影響範囲調査で識別された箇所の慎重な扱い
- リファクタリング範囲を逸脱しない

### テストフェーズ

改修箇所の確認に加え、リグレッション防止のテストを重点的に行います。

- 改修箇所の単体テスト
- 影響範囲のリグレッションテスト
- 既存テスト資産の活用と更新
- 結合テストでの既存業務フローの確認

### リリースフェーズ

既存運用への影響を最小化しながらリリースします。

- リリース手順の確認（停止時間、ロールバック手順）
- 業務側への事前通知
- リリース直後の監視・問い合わせ対応体制の整備

## 6-2. 工程ごとの所要時間の目安

改修案件は、改修規模に応じて工程の比重が変わります。

- **小規模改修（1〜2機能）**：要件20% / 設計30% / 実装20% / テスト25% / リリース5%
- **中規模改修（複数機能・共通処理含む）**：要件15% / 設計35% / 実装20% / テスト25% / リリース5%
- **大規模改修（基盤的な変更）**：要件15% / 設計40% / 実装20% / テスト20% / リリース5%

改修規模が大きくなるほど、設計フェーズ（特に影響範囲調査）の比重が増えます。

# 7章：改修案件で特に重視すべきこと

## 7-1. 影響範囲調査の徹底

改修案件で最も重要なのは、**影響範囲の調査** です。改修箇所だけを見ていると、意図しない既存機能への波及を見落とします。

調査すべき項目は、

- 改修箇所から呼び出される処理・モジュール
- 改修箇所を呼び出す処理・モジュール
- 改修箇所が利用するデータと、それを参照・更新する他箇所
- 改修箇所が利用する共通処理の他での利用箇所

調査方法としては、

- ソースコードの静的解析（grep、IDEの参照検索、解析ツール等）
- 既存設計書の参照
- 有識者へのヒアリング
- データの参照状況のSQL調査

を組み合わせます。詳細は **第1弾 BS-MOD-022** を参照。

## 7-2. 既存資料との整合性の意識

既存システムの設計書とソースが乖離していることは、改修案件で頻繁に発生する問題です。

対応方針として、

- 改修対象に関連する既存設計書の整合状況を最初に確認する
- 設計書とソースの乖離があれば、どちらを正として改修するかを明確にする
- 改修後の設計書の更新方針（更新する/しない）を決める

詳細は **第1弾 BS-MOD-024〜026** を参照。

## 7-3. 既存機能・業務の保証

改修によって既存機能・既存業務が壊れないことの保証が、改修案件の品質の根幹です。

対応方針として、

- リグレッションテストの範囲を明確にする
- 既存テスト資産（過去のテストケース、テストコード等）を活用する
- 業務側との合意を文書化する（変更点と維持する点）
- 改修期間中の業務継続性を確保する

詳細は **第1弾 独自章「既存機能・業務の保証」（BS-MOD-027〜029）** を参照。

## 7-4. 業務側との密な連携

改修は業務側にとって「使い慣れた機能の変更」になることが多く、合意形成が重要です。

- 改修内容を業務側にわかりやすく伝える（画面イメージ、操作手順等）
- 業務側の懸念・要望を引き出す
- 利用者教育の必要性と方法を業務側と共同で検討する
- リリース時の業務影響を業務側と調整する

# 8章：改修案件の典型的な落とし穴と回避策

## 8-1. 影響範囲の見落としによる本番障害

**落とし穴**：改修箇所周辺の影響しか見ず、共通処理経由で他機能に波及する変更を見落として、改修と無関係な既存機能で本番障害が発生する。

**典型的なケース**：
- 共通バリデーション処理を改修した結果、他画面でも入力チェックの挙動が変わってしまった
- 共通日付処理のロジック変更で、別バッチの集計結果が変わってしまった

**回避策**：
- 改修箇所が共通処理を含む場合、その利用箇所を必ず一覧化する
- 共通処理の改修は、別の独立した観点でレビューする
- 影響箇所のリグレッションテストを必ず実施する

## 8-2. 既存設計書の信用しすぎ

**落とし穴**：既存設計書を信用して改修を進めたら、設計書とソースの実装が乖離していて、改修内容が既存仕様と整合しなくなる。

**典型的なケース**：
- 既存設計書には「項目Aは必須」と書かれていたが、実装では任意になっていた。改修で必須前提の処理を追加したが、既存データに任意のレコードが存在し、エラーが多発した。

**回避策**：
- 改修対象の既存設計書とソースの整合状況を最初に確認する
- 重要なロジックは、設計書だけでなくソースで確認する
- 設計書とソースで乖離がある場合、ソースを正として扱うか、業務側に確認する

## 8-3. リファクタリングの過剰拡大

**落とし穴**：改修と一緒にリファクタリングを行ったら、改修と無関係な箇所まで変更が広がり、テスト範囲が膨大になり、リリース後に予期せぬ問題が発生した。

**典型的なケース**：
- 改修ついでに「ここも直そう」と命名規則を統一する変更を入れたら、参照箇所が想定以上に多く、影響範囲が爆発した

**回避策**：
- リファクタリング範囲を明示的に決め、それを超えない
- リファクタリングの目的を文書化する
- 改修と独立してリファクタリングを行う場合、別タスクとして管理する

## 8-4. 既存テスト資産の不在・古さ

**落とし穴**：リグレッションテストをしようとしたが、既存テストコードが古く、現在の実装と動かない。新規にテストを書く時間もない。

**典型的なケース**：
- 数年前に作られた単体テストが、コードベースの変更で動かない
- 結合テストの手順書はあるが、テストデータが現在のDBスキーマと不整合

**回避策**：
- 改修着手前に、既存テスト資産の現状を確認する
- テスト資産の更新が必要な場合、改修工数に組み込む
- 最低限のリグレッションテスト範囲を明確にして、優先的に整備する

## 8-5. リリース時の業務影響の見落とし

**落とし穴**：リリースは技術的には問題なく終わったが、業務側がリリースを把握しておらず、業務開始後に画面が変わって混乱した。

**典型的なケース**：
- リリース手順書には記載があったが、業務側への通知が漏れていた
- 業務マニュアルが更新されていなかった

**回避策**：
- リリース前のチェックリストに「業務側への通知」「業務マニュアル更新」を含める
- リリース直後の問い合わせ対応窓口を準備する
- 重要な変更は、リリース直後に業務側のフォロー（操作説明、Q&A）を行う

## 8-6. 暗黙仕様の調査不足

**落とし穴**：明文化されていない暗黙仕様（特定顧客向けの特別処理、過去の不具合対応の名残等）を見落として改修した結果、特定の業務シナリオで不具合が発生した。

**典型的なケース**：
- 「このコードが入っているレコードだけ別処理する」というロジックが、設計書に書かれていなかった
- 過去の障害対応で入れた一時的な処理が、いつの間にか恒常化していた

**回避策**：
- 改修対象のソースを丁寧に読む
- 有識者（過去の改修担当者、運用担当者）にヒアリングする
- データの実態を調査する（特殊なコード値が存在するか等）

# 11章：リプレイス案件とは

## 11-1. リプレイス案件の定義

リプレイス案件は、既存システムを新しい技術スタック・基盤に置き換える案件です。具体的には以下のようなプロジェクトが該当します。

- 老朽化システムの刷新
- サポート切れOS・ミドルウェアからの移行
- オンプレミスからクラウドへの移行
- レガシー言語（COBOL等）から現代的言語への移行
- モノリシックシステムのマイクロサービス化

## 11-2. リプレイス案件の特徴

リプレイス案件は、以下の特徴を持ちます。

- **業務要件はおおむね不変**：機能・業務はそのまま踏襲することが多い
- **実装・基盤は刷新される**：技術スタックが大きく変わる
- **データ移行が中心テーマ**：既存データを新システムに移す作業が大きな比重
- **機能等価性の確認が重要**：旧システムと同じ振る舞いをすることの保証
- **長期間プロジェクト**：規模が大きく、半年〜数年かかることが多い
- **業務継続性の確保**：プロジェクト期間中も業務は止められない

## 11-3. 典型的なリプレイス案件のパターン

| パターン | 特徴 | 難易度 |
|---------|------|--------|
| 同等機能の技術刷新 | 業務側の影響は最小、技術的な刷新が中心 | 中 |
| 機能改善を含むリプレイス | 業務改善とリプレイスが同時進行 | 高 |
| データ構造の刷新含む | データ移行の難度が大幅に上がる | 高 |
| 段階的リプレイス | 複数フェーズに分割、長期化しやすい | 高 |
| クラウド移行 | インフラ・運用の大幅な変更を伴う | 中〜高 |

# 12章：リプレイス案件の進め方の基本フロー

## 12-1. 工程ごとの進め方

### 要件定義フェーズ

リプレイスの目的・範囲・方針を確定します。

- リプレイスの背景・目的の整理（老朽化、サポート切れ、性能限界等）
- リプレイスの範囲（システム全体か一部か）の確定
- 機能・業務の踏襲・刷新の方針
- 切り替え方式の方針（一括/段階/並行稼働）

### 基本設計フェーズ

旧システムの調査と新システムの設計を並行して進めます。

- 旧システムの全機能・全連携・全運用業務の調査（→ BS-RPL-016、BS-RPL-024）
- 旧データと新データのマッピング（→ BS-RPL-008）
- 暗黙仕様の調査（→ BS-RPL-012）
- データ移行の基本設計（→ 独自章「データ移行」）
- 切り替え戦略の基本設計（→ 独自章「切り替え戦略・並行稼働」）
- 機能等価性検証の方針策定（→ BS-RPL-036）

### 詳細設計フェーズ

実装可能なレベルまで具体化します。

- 旧テーブルと新テーブルの物理レベルマッピング（→ DD-RPL-002）
- 旧実装ロジックの解析と新実装への反映（→ DD-RPL-003）
- 移行ツールの詳細設計（→ DD-RPL-004）
- 旧新比較テストの方針（→ DD-RPL-005）

### 実装フェーズ

新システムの実装と、移行ツールの実装を並行します。

- 新システムの実装（業務ロジック、画面、API等）
- 移行ツールの実装
- 移行ツールの単体テスト・性能テスト

### テストフェーズ

旧新比較テストと、業務シナリオでの検証が中心になります。

- 単体テスト・結合テスト
- 旧新比較テスト（同一データ・同一操作で出力比較）
- 実運用データを使った検証
- 業務担当者による業務観点の検証
- 移行リハーサル（複数回）

### 切り替えフェーズ

入念な計画と段階的な切り替えが重要です。

- 切り替え作業の詳細計画
- 関係者への通知（利用者、連携先、運用担当者）
- 移行リハーサルの最終実施
- 本番データ移行
- 並行稼働期間（採用する場合）
- 旧システムの停止・撤去

### 運用フェーズ

切り替え後の安定化と、旧システムの段階的撤去を行います。

- 切り替え直後の集中監視
- 業務側からの問い合わせ・障害対応
- 並行稼働期間の運用
- 旧システムの撤去（並行稼働終了後）

## 12-2. 工程ごとの所要時間の目安

リプレイス案件は、移行・切り替えの比重が大きくなります。

- **典型的な配分**：要件10% / 設計25% / 実装20% / テスト20% / 移行・切り替え20% / 運用安定化5%

特に **テスト・移行・切り替えで合計40%以上** の時間を確保することが、リプレイス案件成功の鍵です。

# 13章：リプレイス案件で特に重視すべきこと

## 13-1. 旧システムの徹底調査

リプレイス案件の難易度は、旧システムの調査の深さで決まります。

調査すべき項目は、

- 旧システムの全機能（明文化されていないものも含む）
- 旧システムの全連携（突発連携、例外時連携も含む）
- 旧システムの全運用業務（属人的な対応も含む）
- 旧システムのデータ実態（特殊データ、不正データ、暗黙ルール）
- 旧システムの暗黙仕様（特定顧客向け処理、過去の障害対応の名残）

調査が不十分だと、リプレイス後に「旧システムにあったものが新システムにない」という問題が発覚します。

詳細は **BS-RPL-003、BS-RPL-012、BS-RPL-016、BS-RPL-024** を参照。

## 13-2. データ移行の入念な設計と検証

データ移行はリプレイス案件の中心テーマであり、失敗時の影響が最も大きい作業です。

重視すべきこと：

- 移行対象データの範囲を明確に定義する
- 移行方式（一括/段階/差分）を選定する
- 旧データと新データのマッピングを物理レベルまで詳細化する
- 移行データの品質を確保する（クレンジング含む）
- 移行リハーサルを複数回実施する
- 移行失敗時のロールバック方針を準備する

詳細は **第1弾 独自章「データ移行」（BS-RPL-026〜030）** を参照。

## 13-3. 機能等価性の検証

新システムが旧システムと同じ振る舞いをすることを、客観的に検証することが必要です。

検証方法：

- 同一データ・同一操作での出力比較
- 業務シナリオでの検証
- 実運用データを使った検証
- 業務担当者による検証

詳細は **BS-RPL-036、BS-RPL-037、DD-RPL-005** を参照。

## 13-4. 切り替え戦略の慎重な選定

切り替え方式の選定は、業務影響とリスクの両面から検討する必要があります。

- **一括切替**：シンプルだがリスク高、ロールバックが困難
- **段階切替**：機能・業務単位で段階的に切り替え、リスク分散
- **並行稼働**：旧新両方を動かす期間、リスク低いが運用負荷大

選定時の考慮要素：

- 業務停止の許容度
- 切り替え失敗時の影響度
- 並行稼働期間中の運用負荷
- データ同期の複雑さ
- コスト

詳細は **第1弾 独自章「切り替え戦略・並行稼働」（BS-RPL-031〜035）** を参照。

## 13-5. 業務継続性の確保

リプレイス期間中も業務は継続する必要があります。

- プロジェクト期間中に発生する業務（年次処理、定期メンテナンス等）への影響評価
- 並行稼働期間中の業務担当者の負荷増（二重運用）
- 切り替え時の業務停止時間の調整
- 切り替え直後の問い合わせ・障害対応体制

詳細は **BS-RPL-038** を参照。

# 14章：リプレイス案件の典型的な落とし穴と回避策

## 14-1. 旧システムの調査不足による機能欠落

**落とし穴**：旧システムの調査が不十分で、リプレイス後に「旧システムでは出来たことが新システムでは出来ない」という事態が発覚する。

**典型的なケース**：
- 設計書には記載されていない、年に1回しか動かない処理を見落とした
- 特定の顧客だけに適用される特別な処理を実装し忘れた
- 月次・年次バッチの存在を把握していなかった

**回避策**：
- 旧システムの全運用カレンダーを確認する（年間スケジュール）
- 過去のソース変更履歴を調査する
- 旧システム関係者（運用担当者、業務担当者）に丁寧にヒアリングする
- データの実態を調査する（特殊なコード値、特殊なレコードの存在）

## 14-2. データ移行の品質問題

**落とし穴**：移行元データに品質問題（不整合、欠損、不正値等）があるまま移行した結果、新システムに問題が引き継がれる。

**典型的なケース**：
- 旧システムでは許容されていた不正データが、新システムの整合性チェックでエラーになる
- 文字コード変換で文字化けが発生
- NULL値・空文字の扱いの差で、新システムの動作が変わる

**回避策**：
- 移行元データの品質チェックを実施する
- データクレンジングの方針を決める
- 移行リハーサルで品質問題を早期に検出する
- 不正データの取り扱い（補正/除外/エラー扱い）を業務側と合意する

## 14-3. 移行リハーサルの不足

**落とし穴**：移行リハーサルを十分に行わず、本番移行で想定外のトラブルが発生し、許容時間を超過する。

**典型的なケース**：
- リハーサルでは少量データで成功したが、本番の大量データで処理時間が想定の数倍かかった
- リハーサル環境と本番環境の差異が原因で、本番で予期せぬエラーが発生
- 移行ツールの不具合がリハーサルで見つからず、本番で発覚

**回避策**：
- リハーサルを複数回実施する（最低2〜3回）
- 本番相当のデータ量・環境で実施する
- リハーサル時に実測した所要時間を本番計画に反映する
- リハーサルで発見した問題を必ず潰してから本番に臨む

## 14-4. 暗黙仕様の見落とし

**落とし穴**：旧システムの設計書に書かれていない暗黙仕様を見落とし、リプレイス後の業務影響として現れる。

**典型的なケース**：
- 特定の業務シナリオでだけ動く特殊処理を見落とした
- 過去の障害対応で入れた一時的な処理が、運用上は必須になっていた
- 業務担当者が当たり前に使っていたが、ドキュメント化されていない機能があった

**回避策**：
- 旧システムのソースを丁寧に読む（特に分岐の多い箇所）
- 業務担当者にヒアリングし、業務手順を実際に観察する
- 「設計書に書かれていない仕様」リストを作って、新システムへの反映方針を決める

詳細は **BS-RPL-012** を参照。

## 14-5. 並行稼働期間の運用負荷の見積もり不足

**落とし穴**：並行稼働期間中の運用負荷を軽く見積もり、業務担当者の負担が想定以上になり、業務に支障をきたす。

**典型的なケース**：
- 並行稼働期間中、業務担当者が旧新両方のシステムでデータ入力する必要があった
- 旧新システムで結果が異なる事象が頻発し、確認・調整に時間を取られた
- 並行稼働期間が当初予定より長引き、業務側のモチベーションが低下

**回避策**：
- 並行稼働期間の運用フローを業務側と一緒に詳細化する
- 二重運用の負荷を定量的に見積もる
- 並行稼働期間の終了条件を明確にする
- 並行稼働期間中の運用体制を強化する（増員、業務側の負荷軽減策）

## 14-6. 切り替え失敗時のロールバック準備不足

**落とし穴**：切り替えが想定通りに進まず、ロールバックを判断したが、ロールバック手順が不十分で旧システムへの戻しに長時間かかる。

**典型的なケース**：
- ロールバック手順を書面で用意していたが、リハーサルしておらず、実際にやると問題が発生
- データ移行を完了した状態でロールバックしようとしたが、旧データの状態を戻す手段がなかった
- ロールバック判断の基準が曖昧で、判断が遅れた

**回避策**：
- ロールバック手順を文書化する
- ロールバックリハーサルを実施する
- ロールバック判断の基準・権限・タイミングを事前に決める
- 切り替え判断のチェックポイント（ゲート）を設ける

詳細は **BS-RPL-030、BS-RPL-034** を参照。

## 14-7. 連携先との調整不足

**落とし穴**：外部連携先との調整が不十分で、新システム稼働後に連携トラブルが発生する。

**典型的なケース**：
- 連携先のシステムが旧I/F仕様を前提にしており、新I/Fに対応する時間がなかった
- 連携先の本番切り替えタイミングと自システムの切り替えタイミングが噛み合わなかった
- 連携テストが連携先の都合で十分に実施できなかった

**回避策**：
- 連携先との調整を早期に開始する
- 連携先ごとの対応スケジュールを管理する
- 連携先のテスト環境利用可否を早期に確認する
- 並行稼働期間中の連携運用を綿密に計画する

# 17章：新規案件とは

## 17-1. 新規案件の定義

新規案件は、ゼロから新しいシステムを構築する案件です。具体的には以下のようなプロジェクトが該当します。

- 既存業務をシステム化する初めてのプロジェクト
- 新規事業の立ち上げに伴うシステム構築
- 既存システムとは別の新しいサービスの構築
- 業務プロセスの新規構築を伴うシステム開発

## 17-2. 新規案件の特徴

新規案件は、以下の特徴を持ちます。

- **既存の制約がない**：既存システム・既存データの制約を受けない
- **業務プロセスから設計する**：業務側との合意形成が中心テーマ
- **規約・標準を整備する必要がある**：参照すべき既存規約がない
- **初期データの整備が必要**：既存データを引き継げない
- **利用者教育の比重が大きい**：既存業務の延長ではないことが多い
- **要件の変動リスクが大きい**：要件が固まりきらないまま進むことがある

## 17-3. 典型的な新規案件のパターン

| パターン | 特徴 | 難易度 |
|---------|------|--------|
| 既存業務の単純システム化 | 業務プロセスは確立されている | 低〜中 |
| 業務プロセス改善を伴う | 業務側との合意形成が重要 | 中 |
| 新規事業立ち上げ | 業務自体が固まっていない、要件変動大 | 高 |
| 段階的リリース | スコープを段階的に広げる、計画が複雑 | 中〜高 |
| 全社共通基盤の新規構築 | 多数の利用部門との調整、影響大 | 高 |

# 18章：新規案件の進め方の基本フロー

## 18-1. 工程ごとの進め方

### 要件定義フェーズ

業務要件と業務プロセスの確定が中心です。

- ビジネス目的の明確化
- 想定利用者・利用シーンの定義
- 業務プロセスの設計（業務フロー、関係者、業務ルール）
- 業務側との合意形成
- 要件の優先順位付け（特に段階リリースの場合）

### 基本設計フェーズ

業務プロセスをシステムに落とし込み、設計します。

- 業務プロセスとシステム機能の対応付け（→ BS-NEW-019）
- 業務要件から機能への落とし込み（→ BS-NEW-008）
- データ構造の設計（業務分析からエンティティを導出）（→ BS-NEW-007）
- 段階リリース計画（採用する場合）（→ BS-NEW-003、BS-NEW-009）
- セキュリティ基準の選定（→ BS-NEW-014）
- 運用体制の構築計画（→ BS-NEW-016）

### 詳細設計フェーズ

実装可能なレベルまで具体化します。新規案件特有の論点として、規約整備があります。

- 詳細設計の標準・規約の整備（命名規則、コーディング規約等）（→ DD-NEW-001）
- フレームワーク・ライブラリの利用方針（→ DD-NEW-002）
- 開発環境・ビルド構成の詳細設計（→ DD-NEW-003）
- テスト基盤の整備計画（→ DD-NEW-004）

### 実装フェーズ

設計書と規約に基づいて実装を行います。

- 規約に準拠した実装
- フレームワーク標準のパターンに従う
- 開発環境・ビルド・CI/CDの整備と並行

### テストフェーズ

参照すべき既存テスト資産がないため、テスト基盤・テストデータの整備から始めます。

- テストデータの設計と作成
- テスト基盤の構築
- 単体・結合・システムテストの実施
- 業務担当者による受入確認

### 初期データ整備フェーズ

新規案件特有のフェーズです。稼働開始時に必要なデータを準備します。

- 初期データ・マスタデータの作成
- データ品質の確保
- データ投入のリハーサル
- 業務側との連携（データ作成、検証）

### リリース・展開フェーズ

利用者への展開と教育が重要なフェーズです。

- 利用者教育の実施
- 業務マニュアルの整備
- 段階リリースの場合、各段階の計画通りの実施
- 稼働後のフォロー体制

## 18-2. 工程ごとの所要時間の目安

新規案件は、要件定義・業務プロセス設計の比重が大きくなります。

- **典型的な配分**：要件25% / 設計25% / 実装25% / テスト15% / リリース・展開10%

特に **要件と設計（業務プロセス含む）で合計50%** を確保することが、新規案件成功の鍵です。

# 19章：新規案件で特に重視すべきこと

## 19-1. 業務プロセスの設計と業務側合意

新規案件では、業務プロセス自体を設計する必要があります。

重視すべきこと：

- 業務プロセスの全体像（業務フロー、関係者、業務ルール、判断ポイント）を定義する
- 業務プロセスとシステム機能の対応関係を明確にする
- 業務側との合意を文書化する
- 既存業務との変更点を業務側に明示する

詳細は **第1弾 独自章「業務プロセス構築」（BS-NEW-018〜021）** を参照。

## 19-2. 規約・標準の事前整備

新規案件では参照すべき既存規約がないため、最初に規約・標準を整備する必要があります。

整備すべき項目：

- 命名規則（クラス名、メソッド名、変数名、テーブル名等）
- コーディング規約
- 詳細設計書のテンプレート
- ドキュメント体系
- バージョン管理規則
- コミットメッセージ規約
- レビュープロセス

これらを設計フェーズの早い段階で整備しないと、設計書・実装が設計者ごとにバラバラになります。

詳細は **DD-NEW-001** を参照。

## 19-3. 初期データの整備計画

稼働開始時に必要となる初期データ・マスタデータの整備は、新規案件特有の重要タスクです。

重視すべきこと：

- 必要となる初期データの範囲と内容を定義する
- データ整備の役割分担（業務側、開発側）を明確にする
- 稼働日から逆算したスケジュールを立てる
- データ品質を確保する
- 投入方式と投入リハーサルを計画する

詳細は **第1弾 独自章「初期データ整備」（BS-NEW-022〜026）** を参照。

## 19-4. 段階的リリースの計画

新規案件は規模が大きくなりやすく、段階的にリリースする方が成功しやすいことがあります。

段階リリースを検討する際の重視点：

- 各段階で提供する機能スコープを明確にする
- 未提供機能の代替手段（手作業運用等）を準備する
- 段階間の機能の連続性を保つ
- 段階ごとに運用業務を立ち上げる

詳細は **BS-NEW-003、BS-NEW-009、BS-NEW-012、BS-NEW-017** を参照。

## 19-5. 利用者教育・展開計画

新規システムは利用者にとって新しい業務であり、習熟が必要です。

重視すべきこと：

- 教育対象者・教育内容・教育方法を計画する
- 業務マニュアルを整備する
- 稼働後のフォロー体制（問い合わせ窓口、補足教育）を準備する
- 段階リリースの場合、各段階で必要な教育を計画する

詳細は **BS-NEW-021** を参照。

# 20章：新規案件の典型的な落とし穴と回避策

## 20-1. 業務プロセスの未確定のまま設計開始

**落とし穴**：業務プロセスが業務側と合意できていないまま設計を進めた結果、設計やり直しが頻発する。

**典型的なケース**：
- 業務側の要望が次々と追加され、設計が固まらない
- 業務側の意思決定者が複数おり、合意が取れない
- 「とりあえず動けば良い」と進めたが、業務での実用性に欠けるシステムになった

**回避策**：
- 業務プロセスを文書化し、業務側との合意を取る（口頭合意で進めない）
- 業務側の意思決定者を明確にする
- 業務フロー図やプロトタイプを使って、業務側に視覚的に確認してもらう
- 要件凍結のタイミングを設ける

## 20-2. 規約整備の後追いによる設計のばらつき

**落とし穴**：規約を整備しないまま設計・実装を始めた結果、設計書・実装が設計者ごとにバラバラになり、後から修正できない。

**典型的なケース**：
- 命名規則がないため、テーブル名・カラム名が機能ごとに異なる流派になる
- コーディング規約がないため、ソースコードのスタイルが人によって違う
- 詳細設計書のテンプレートがないため、書き方が設計者ごとに異なる

**回避策**：
- プロジェクト立ち上げ時に最低限の規約を整備する
- 詳細設計書のテンプレートを事前に作る
- 規約準拠の確認をレビューに組み込む
- 規約違反を早期に発見・修正する

## 20-3. 初期データ整備の遅れによる稼働遅延

**落とし穴**：初期データの整備計画が甘く、稼働直前に「データが揃わない」事態になる。

**典型的なケース**：
- マスタデータの作成を業務側に依頼したが、業務側のリソース不足で進まない
- データの作成方法（フォーマット、入力ルール）が曖昧で、作り直しが発生
- 稼働直前にデータ品質問題が発覚し、稼働延期

**回避策**：
- 初期データの整備を早期に計画する（要件定義・基本設計のうちに）
- 業務側の作業負荷を見積もり、業務側の体制と整合させる
- データ作成のテンプレート・ルールを明確にする
- データ投入リハーサルを実施する

## 20-4. 段階リリース時の機能スコープ曖昧

**落とし穴**：段階リリースの方針はあるが、各段階のスコープが曖昧で、リリース時に「使えない機能がある」状態になる。

**典型的なケース**：
- 第1段階で提供すべき機能と、第2段階に回す機能の境界が曖昧
- 第1段階リリース時に、想定していなかった機能の不足が発覚
- 段階間でデータの引き継ぎ方法が決まっていなかった

**回避策**：
- 各段階で提供する機能を明示的に定義する
- 各段階の業務シナリオを通しで検証する
- 未提供機能の代替手段を計画する
- 段階間のデータ・運用の引き継ぎ方法を設計する

## 20-5. 利用者教育の不足による稼働後の混乱

**落とし穴**：利用者教育が不十分で、稼働後に利用者がシステムを使いこなせず、業務が滞る。

**典型的なケース**：
- 業務マニュアルは作ったが、実際の操作トレーニングをしていなかった
- 教育を一度実施したが、稼働まで時間が空いて利用者が忘れていた
- 教育対象者の選定に漏れがあった

**回避策**：
- 教育計画を稼働日から逆算して立てる
- 操作トレーニング（ハンズオン）を実施する
- 稼働直前にも復習教育を行う
- 稼働後の問い合わせ窓口・補足教育を準備する

## 20-6. 性能要件の根拠不明

**落とし穴**：性能要件を業務側の希望そのまま設定したが、根拠が曖昧で、稼働後に過剰または過小であることが発覚する。

**典型的なケース**：
- 「同時利用者100人」と要件定義したが、実際は10人程度しか使わずオーバースペック
- 想定利用者数を控えめに見積もりすぎて、稼働後の増加に対応できない

**回避策**：
- 性能要件の根拠（想定利用者数、業務上の許容時間等）を文書化する
- 算出ロジックを明示する
- 想定が外れた場合の対応方針を準備する

詳細は **BS-NEW-013** を参照。

## 20-7. 運用体制の構築遅れ

**落とし穴**：運用体制の構築が遅れ、稼働開始時に運用担当者がいない、または準備不足の事態が発生する。

**典型的なケース**：
- 運用担当者の確保が稼働直前になった
- 運用担当者への教育・引き継ぎが不十分
- 運用ツールの整備が間に合わなかった

**回避策**：
- 運用体制の構築計画を早期に立てる
- 運用担当者の確保・育成を計画的に進める
- 運用ドキュメントの整備時期を計画する
- 開発チームから運用チームへの引き継ぎ計画を立てる

詳細は **BS-NEW-016、BS-COM-080** を参照。
